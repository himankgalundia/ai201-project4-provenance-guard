# Provenance Guard

A content-attribution service for a publishing platform. A creator submits a
piece of text; Provenance Guard estimates whether it was **written by a human**
or **generated by AI**, returns a **calibrated confidence score** and a
**plain-language transparency label** for readers, logs every decision in a
**structured audit log**, and lets creators **appeal** a classification.

See [`planning.md`](planning.md) for the full design spec (architecture diagram,
signal design, uncertainty model, label copy, appeals workflow, edge cases, and
the AI tool plan).

## Architecture

```
                         [rate limiter: 10/min, 100/day]
client --(text, creator_id)--> POST /submit
                                   |
                         assign content_id (UUID) ; store record (status=classified)
                                   |
                 +-----------------+------------------+
                 v                                    v
   Signal 1: Groq LLM                    Signal 2: stylometric heuristics
   --(text)-->  llm_score [0..1]          --(text)-->  stylometric_score [0..1]
                 |                                    |
                 +-----------------+------------------+
                                   v
              ai_score = 0.65*llm + 0.35*stylometric   (combined score)
                                   v
        attribution + confidence  (thresholds 0.35 / 0.65)
                                   v
                 transparency label (Variant A / B / C)
                                   v
        audit-log entry  {scores, attribution, confidence, status}
                                   v
   response {content_id, attribution, confidence, signals, label, status}

APPEAL: POST /appeal (content_id, creator_reasoning)
        -> status: classified -> under_review ; store reasoning
        -> audit-log 'appeal' entry beside the original decision
        -> response {status: under_review, appeal_logged: true}
```

The codebase is three modules: [`signals.py`](signals.py) (both detection
signals + scoring + label generation), [`db.py`](db.py) (SQLite content store +
audit log), and [`app.py`](app.py) (Flask routes + rate limiting).

## Web UI

In addition to the JSON API, the service ships a lightweight reader/creator-facing
**web UI** served at `http://localhost:5000/` (template:
[`templates/index.html`](templates/index.html), no build step or external
dependencies — it just calls the same `/submit`, `/appeal`, and `/log` endpoints
via `fetch`). This is the optional visual mockup of the transparency label the
assignment mentions; the canonical label copy is still the written text below.

It has four parts:
- **Submit content** — a text box + creator ID, with two "Try an example" buttons
  (AI-ish / human-ish) that pre-fill the box.
- **Transparency label** — the actual reader-facing label, **color-coded**
  (red = AI, green = human, amber = uncertain) with a **confidence meter bar**.
- **Signal breakdown** — bars for the LLM score, the stylometric score, and the
  combined AI score, plus the LLM's plain-text rationale — so the two independent
  signals are visible side by side.
- **Contest this classification** — an appeal box that posts to `/appeal`, flips
  status to `under_review`, and shows a confirmation banner.
- **Audit log** — a live table (colored pills for attribution/status) that
  refreshes after every submission and appeal.

The UI is purely a presentation layer over the API; all detection, scoring, rate
limiting, and logging happen server-side in exactly the same code paths the
`curl`/JSON callers hit.

## Detection signals (multi-signal pipeline)

Two **genuinely independent** signals — one *semantic*, one *structural*. They
can disagree, and that disagreement pushes borderline content into "uncertain"
rather than a false verdict.

### Signal 1 — LLM classifier (Groq `llama-3.3-70b-versatile`) — *semantic*
Sends the text to the model with a forensic prompt and `response_format=json`,
getting back `ai_probability ∈ [0,1]` plus a one-sentence rationale. **Captures**
holistic coherence, tone, and the subtle over-smoothness LLMs produce — patterns
no single statistic captures. **Why this signal:** a large model has seen huge
amounts of both human and AI text and is the single most accurate "does this read
as machine-written?" judge available for free. **Blind spot:** it is itself an
LLM — non-deterministic, can be confidently wrong, penalizes formal human prose,
and is a network dependency. If the call fails it returns a neutral `0.5` flagged
`degraded`, and the blend re-weights to `0.20/0.80` so a failed call cannot fake
a verdict.

### Signal 2 — Stylometric heuristics (pure Python) — *structural*
An internal ensemble of four weighted sub-metrics, each mapped to an AI-likeness
contribution in `[0,1]`:

| Sub-metric | Weight | What it measures | Direction |
|---|---|---|---|
| **Burstiness** (sentence-length coefficient of variation) | 0.40 | rhythm variety | low variance ⇒ AI |
| **AI-cliché density** ("furthermore", "it is important to note", "delve"…) | 0.30 | formulaic connective use | high ⇒ AI |
| **Informality markers** (contractions, lowercase starts, "ok so", "honestly") | 0.20 | casual human voice | high ⇒ human |
| **Lexical diversity** (type-token ratio) | 0.10 | vocabulary repetition | very low ⇒ AI |

**Why this signal:** it's transparent, deterministic, always-available, and
measures something the LLM doesn't — the literal statistical shape of the prose.
**Blind spot:** no understanding of meaning. Repetitive simple poetry looks
"uniform" and scores AI-like; an AI told to "write casually with typos" can spoof
the informality markers. Short inputs give unstable variance, so the classifier
down-weights inputs under 40 characters toward "uncertain".

### Combination
```
ai_score = 0.65 * llm_score + 0.35 * stylometric_score      # (0.20/0.80 if LLM degraded)
```
The LLM carries the larger weight (more accurate, holistic); the stylometric
signal is an independent cross-check that can pull a borderline LLM call back
toward uncertain.

## Confidence scoring with uncertainty

`ai_score ∈ [0,1]` is the estimated probability the text is AI. **Attribution**
comes from two thresholds, and the reported **confidence = `max(ai_score,
1 − ai_score)`** — the probability of whichever class was chosen, ranging
**0.50 → 1.00**:

| `ai_score` | attribution | label variant |
|---|---|---|
| `>= 0.65` | `likely_ai` | high-confidence AI |
| `0.35 – 0.65` | `uncertain` | uncertain |
| `<= 0.35` | `likely_human` | high-confidence human |

This is deliberately **not a binary flip at 0.5**. A `0.51` confidence (`ai_score
≈ 0.51`) lands in the *uncertain* band and shows the uncertain label; a `0.95`
confidence shows a definitive label. The confidence number degrades smoothly
toward 0.50 as the two signals weaken or disagree.

### How I tested that the scores are meaningful
`test_scoring.py` runs both signals **separately** plus the combined classifier
on four labeled inputs spanning the range (clear AI, clear human, formal human,
lightly-edited AI). Printing each signal independently makes a misbehaving signal
obvious. Observed results (real run):

| Input | llm_score | stylometric_score | combined ai_score | attribution | confidence |
|---|---|---|---|---|---|
| **Clearly AI** | 0.90 | 0.757 | **0.850** | likely_ai | 0.85 |
| **Clearly human** | 0.20 | 0.071 | **0.155** | likely_human | 0.845 |
| **Formal human** (borderline) | 0.80 | 0.555 | **0.714** | likely_ai | 0.71 |
| **Lightly-edited AI** (borderline) | 0.70 | 0.254 | **0.544** | uncertain | 0.544 |

The clear cases sit at opposite ends (0.85 vs 0.155); the two borderline cases
land in the middle exactly as designed — the lightly-edited AI reaches the
**uncertain** band, and the formal-human case is flagged AI at a *moderate* 0.71
(not 0.95), which is precisely the documented false-positive scenario the appeal
workflow exists for.

### Two example submissions (high vs lower confidence)

**High-confidence (clearly AI), confidence 0.84:**
```json
{ "attribution": "likely_ai", "confidence": 0.84,
  "signals": { "llm_score": 0.9, "stylometric_score": 0.7285,
               "combined_ai_score": 0.84 },
  "label": { "variant": "high_confidence_ai" } }
```

**Lower-confidence (lightly-edited AI), confidence 0.544:**
```json
{ "attribution": "uncertain", "confidence": 0.544,
  "signals": { "llm_score": 0.7, "stylometric_score": 0.254,
               "combined_ai_score": 0.544 },
  "label": { "variant": "uncertain" } }
```
The signals *disagree* in the second case (LLM leans AI, stylometrics lean
human) — the blend correctly reports genuine uncertainty rather than picking a
side.

## Transparency label — all three variants

The label returned by `/submit` changes with the confidence score. `{pct}` is the
confidence as a percentage.

**Variant A — High-confidence AI** (`attribution = likely_ai`):
> 🤖 **Likely AI-Generated** — Our analysis suggests this content was most likely
> produced by an AI system (confidence: {pct}%). Both our language-model check and
> our writing-style check pointed toward automated generation. This is an
> automated estimate, not a certainty. If you wrote this yourself, you can appeal.

**Variant B — High-confidence human** (`attribution = likely_human`):
> ✍️ **Likely Human-Written** — Our analysis suggests this content was most likely
> written by a person (confidence: {pct}%). The writing showed the natural
> variation and voice typical of human authors. This is an automated estimate and
> may be wrong.

**Variant C — Uncertain** (`attribution = uncertain`):
> ❓ **Attribution Uncertain** — We could not confidently determine whether this
> content was written by a person or generated by AI (confidence: {pct}%). Our two
> checks were borderline or disagreed with each other. Please treat the origin of
> this content as unknown.

All three are reachable in the test run above (high_confidence_ai,
high_confidence_human, and uncertain all appear).

## Appeals workflow

`POST /appeal` with `{ content_id, creator_reasoning }`:
- validates the `content_id` exists (`404` if not),
- updates the content record's `status` to **`under_review`** and stores the
  reasoning,
- appends an **`appeal`** audit entry that carries the original attribution,
  confidence, and both signal scores **beside** the creator's reasoning, so a
  human reviewer sees both at once,
- returns a confirmation. **No automated re-classification** — a person makes the
  final call.

Live example:
```bash
curl -s -X POST http://localhost:5000/appeal -H "Content-Type: application/json" \
  -d '{"content_id":"a108e722-...","creator_reasoning":"I wrote this myself..."}'
```
```json
{ "appeal_logged": true, "content_id": "a108e722-...",
  "status": "under_review",
  "original_decision": { "attribution": "likely_human", "confidence": 0.8653 },
  "message": "Your appeal has been received. This content is now under review..." }
```

## Rate limiting

Flask-Limiter (in-memory storage) on `/submit`:

```python
@limiter.limit("10 per minute;100 per day")
```

**Reasoning for these values:** a genuine creator submits their *own* work — a
handful of pieces in a sitting, not hundreds. **10/minute** comfortably covers a
writer pasting and re-submitting a few drafts while editing, but stops a script
from flooding the endpoint (each submit triggers a paid Groq call, so this also
caps cost/abuse). **100/day** bounds sustained automated scraping without
inconveniencing any realistic single author. `/appeal` is limited more loosely
(`20/min; 200/day`) since contesting is low-volume and shouldn't be throttled as
aggressively as classification. Storage is `memory://` for local dev; production
would use Redis so limits hold across processes.

**Evidence** — 12 rapid requests against the 10/min limit (the first 2 of the
minute's budget were already spent by earlier test submissions, so the cap is hit
at request 9):
```
request 1 -> 200      request 7  -> 200
request 2 -> 200      request 8  -> 200
request 3 -> 200      request 9  -> 429
request 4 -> 200      request 10 -> 429
request 5 -> 200      request 11 -> 429
request 6 -> 200      request 12 -> 429
```
A `429` returns `{ "error": "rate_limit_exceeded", "message": "..." }`.

## Audit log

Every classification and every appeal is written to the `audit_log` table in
SQLite as a structured row (`content_id`, `event_type`, `timestamp`, JSON
`details`). `GET /log` returns them newest-first. A real `classification` entry
records both individual signal scores, the combined score, attribution,
confidence, label variant, degraded flag, and status; an `appeal` entry records
the creator's reasoning beside the original decision.

## Stretch feature — ensemble detection

The stylometric signal is itself a **documented weighted ensemble of four
sub-metrics** (burstiness 0.40, cliché density 0.30, informality 0.20, lexical
diversity 0.10), and that result is then blended with the LLM signal under a
second documented weighting (0.65 / 0.35, or 0.20 / 0.80 when the LLM is
degraded). So the final decision draws on five distinct measurements combined by
an explicit voting/weighting scheme. Every component score is surfaced in both
the `/submit` response and the audit log, so the ensemble is fully inspectable.

## Known limitations

- **Formal human prose is the dominant false-positive.** Academic, legal, or
  financial writing is long, uniform, and cliché-heavy — it trips *both* signals
  toward AI. In testing, a real human paragraph on monetary policy scored 0.71
  (`likely_ai`). This is structural to the signals: the stylometric heuristics
  literally measure uniformity, and "smooth formal prose" is exactly what they
  flag. Mitigations are the deliberately moderate confidence (0.71, not 0.95) and
  the appeal workflow — but the misclassification itself is expected.
- **Repetitive simple poetry** (refrains, short plain lines) looks structurally
  AI-like (low variance, low lexical diversity) even when human-authored.
- **Lightly human-edited AI** is genuinely hard; the system aims to land these in
  "uncertain" rather than confidently "human", which is the safest failure mode
  but not a true detection.
- The LLM signal is **non-deterministic and a network dependency**; identical
  text can score slightly differently across runs, and an outage degrades the
  system to stylometrics-only.

## Spec reflection

**How the spec helped:** writing the uncertainty model in `planning.md` *before*
coding forced me to define `confidence = max(ai_score, 1 − ai_score)` and the
0.35/0.65 thresholds as concrete numbers. When I generated the scoring function
with AI help, I could check it line-by-line against those exact thresholds — and
it had initially flipped a comparison, which I caught immediately because the
spec said precisely what each band should do.

**How the implementation diverged:** the spec didn't anticipate the *degraded
LLM* case in detail. While building Signal 1 I added a fallback (neutral 0.5 +
re-weighting to 0.20/0.80) so a Groq outage can't masquerade as a confident
semantic verdict. I also added a short-input dampener (inputs under 40 chars get
pulled toward 0.5) after realizing burstiness is meaningless on one sentence —
neither was in the original spec, but both follow directly from the blind spots I
*had* documented.

## AI usage

1. **Stylometric signal + scoring (M4).** I gave the AI tool the detection-signal
   and uncertainty sections of `planning.md` and asked it to generate the
   stylometric function and the `combine_and_classify` logic. Its first draft
   used reasonable-looking but *unspecified* thresholds (a flat 0.5 split) and
   normalized burstiness on a different scale than I'd planned. I overrode the
   thresholds to the documented 0.35/0.65 bands, re-derived the CV→AI mapping so
   it matched my "CV ≤ 0.2 ⇒ AI, ≥ 0.7 ⇒ human" spec, and added explicit
   per-sub-metric weights so the ensemble was inspectable.

2. **Flask skeleton + appeal endpoint (M3/M5).** I asked the AI tool to scaffold
   the Flask app, `/submit`, and `/appeal` from the architecture diagram. It
   produced a working skeleton but stored everything in an in-memory dict; I
   replaced that with the SQLite layer in `db.py` so the audit log survives
   restarts and is queryable, and I added the `429` JSON error handler and the
   `degraded`/`short_input` flags into the logged details, which the generated
   version omitted.
